系统概览 & Dual Socket

SYS-221HE-FTNR 是完整的 2U 服务器系统，X13DEM 是其中的主板。面试最常问为什么要用双路——答案要从三个维度展开。

系统 vs 主板

型号是什么角色
SYS-221HE-FTNR完整 2U 服务器系统Short-depth, front-I/O，定义机箱布局和外部接口
X13DEM主板 (11.5" × 17")定义所有核心硬件能力，同一块板也用于 SYS-121H / 621H 等

为什么要 Dual Socket

KEY双路不是"更高级"，而是 workload 同时需要更多 CPU + 更大内存 + 更多 I/O 三个维度时的选择。
维度单路上限双路扩展X13DEM 实际值
算力1× CPU, max 64C2× CPU最多 128 cores
内存16 DIMM / 8ch / ~4TB32 DIMM / 16ch最多 8TB (3DS RDIMM)
PCIe I/O~80 lanes~160 lanesPCIe 5.0 全速
Q面试怎么一句话解释 dual-socket？▶
标准答法
A dual-socket design is used when the workload requires more compute, memory, and I/O than a single CPU can provide. This platform scales to 128 cores, 8 TB memory, and roughly 160 PCIe 5.0 lanes by using two processors.

应用场景

Telecom / NFV / 5G
需要多核处理大量虚拟化网络功能，同时需要高带宽 NIC 直连 CPU
AI Inference Edge
多 GPU 卡需要大量 PCIe 5.0 x16 slot，内存带宽需求大
数据库 / In-memory
需要超大内存容量（TB级），单路无法满足
HPC / Scientific
高核心数 + 高内存带宽，NUMA-aware 应用可充分利用

UPI 的作用

Socket 间通信
CPU1 和 CPU2 通过 UPI 链路直连，可以互相访问对方的内存（cross-socket）
Cache Coherency
两颗 CPU 看到一致的内存视图。CPU2 写入的数据，CPU1 能读到最新版本

NUMA 是什么

KEYUPI 的存在引出了 NUMA（Non-Uniform Memory Access）——本地内存快，跨 socket 内存慢。
访问类型路径延迟
本地内存 (CPU1 → 其自己的 DIMM)CPU 内部 Memory Controller低（约 60–80 ns）
远端内存 (CPU1 → CPU2 的 DIMM)CPU1 → UPI → CPU2 Memory Controller → DRAM高（额外 +40–80 ns）
QQPI 和 UPI 什么关系？怎么回答？▶
标准答法
QPI was Intel's older socket interconnect. For Sapphire Rapids and Emerald Rapids, it has been replaced by UPI 2.0 at 16 GT/s with 3 to 4 links. Same concept — coherent socket-to-socket communication — but newer and faster generation.
QNUMA 是什么？对实际应用有什么影响？▶
标准答法
NUMA means each CPU has its own local memory, which is fast to access. Accessing the other socket's memory requires going through UPI, which adds latency. The OS scheduler tries to keep a process and its memory on the same NUMA node for best performance. Applications can also be NUMA-aware to pin workloads explicitly.

Memory Controller 在哪里？

!Memory Controller 集成在 CPU 内部，不是独立芯片，不是 Chipset，也不是 BMC。面试容易混淆。
为什么集成在 CPU
减少内存访问延迟，避免走外部总线。CPU 直接通过内存通道读写 DRAM。
为什么要 RDIMM
Register Buffer（寄存器缓冲）减少电气负载，支持更多 DIMM / 通道，提升信号完整性。消费级用 UDIMM，无法扩展到服务器规模。
Q1DPC 和 2DPC 有什么区别？什么时候选哪个？▶
标准答法
1DPC gives higher speed at DDR5-5600, but lower total capacity per system. 2DPC doubles the DIMM count to 32 slots and can reach 8 TB, but the speed drops to DDR5-4400 due to increased electrical load on the memory bus. You choose 2DPC when the workload is memory-capacity bound, and 1DPC when you prioritize memory bandwidth.

为什么 M.2 只有 PCIe 3.0？

设计意图
M.2 用于 boot/OS drive，不是性能存储。限制到 PCIe 3.0 x2 可以把高速 PCIe 5.0 lanes 留给 NVMe 数据盘和 GPU。
带宽够用
PCIe 3.0 x2 ≈ 2 GB/s，对于 OS 盘的顺序读 + 随机小 IO 已经足够，无需浪费高速 lanes。

NVMe / SATA / SAS

三种存储接口是面试必考对比。重点：NVMe 直连 CPU，SATA 走 chipset，SAS 需要 HBA card。

三种接口对比总览

接口速率路径控制器用途
NVMePCIe 5.0 x8 ≈ 32 GB/sCPU 直连 (MCIO)无需额外 controller高性能数据存储
SATA III6 Gbps ≈ 600 MB/sC741 → DMI → CPUC741 chipset 内建boot drive, 低成本存储
SAS-422.5 Gbps / lanePCIe HBA → CPU需要 PCIe add-in HBA企业 HDD, JBOD 扩柜
!X13DEM 没有板载 SAS。如果 workload 需要 SAS，必须在 PCIe slot 插一张 SAS HBA 或 RAID controller。

NVMe 详解

属性说明
连接方式8× MCIO connectors，每个 PCIe 5.0 x8
最高单盘带宽~32 GB/s (PCIe 5.0 x8)
路径CPU PCIe Root Port → MCIO → NVMe SSD，无中间控制器
RAID通过 Intel VROC key header 做 NVMe RAID，不需要 HBA

SATA 的瓶颈在哪里

SATA → DMI 瓶颈
SATA 走 C741 → DMI 4.0 → CPU。DMI 4.0 总带宽约 16 GB/s，多个 SATA 设备共享这条总线。
实际使用场景
适合 boot drive、日志盘、低成本归档存储。对于数据库或高 IOPS 应用，NVMe 才是正确选择。

SAS HBA 的作用

功能说明
协议转换把 PCIe 协议转成 SAS/SATA 协议，驱动 SAS 设备
可选 RAID部分 HBA 支持硬件 RAID 0/1/5/6/10
SAS Expander可连接 SAS expander，扩展到数十甚至上百个盘
向下兼容SAS controller 通常也能驱动 SATA 盘
QNVMe 和 SATA 有什么核心区别？▶
标准答法
NVMe connects directly to CPU PCIe lanes through MCIO connectors, giving up to 32 GB/s per drive on this platform. SATA goes through the chipset at only 6 Gbps, and is also limited by the shared DMI link to the CPU. NVMe is the high-performance storage path; SATA is for lower-cost or legacy drives like boot media.
Q为什么 SAS 需要额外 controller？▶
标准答法
SAS is not natively supported by the CPU or chipset on this platform. A SAS HBA or RAID controller in a PCIe slot acts as the protocol bridge, translating PCIe to SAS. It also optionally provides hardware RAID and supports connecting SAS expanders for large external JBOD enclosures.

网络 & AIOM

AIOM 是 Supermicro 对 OCP 3.0 的称呼，核心是模块化 NIC 直连 CPU PCIe，适合高带宽网络场景。

网络架构

接口规格连到哪里用途
2× AIOM slotPCIe 5.0 x16, OCP 3.0 supersetCPU 直连生产业务网络
1× BMC RJ451GbE dedicatedAST2600 BMC带外管理网络

AIOM 的优势

模块化选择
根据需求插不同速率的 NIC 模块：1G / 10G / 25G / 100G / 200G，不改主板
节省 PCIe slot
AIOM 不占用标准 PCIe expansion slot，让 4个 x16 slot 全部留给 GPU / HBA
直连 CPU
AIOM 走 CPU-side PCIe 5.0，适合 NFV / 5G 场景的高带宽低延迟网络需求
OCP 标准
兼容 OCP NIC 3.0 标准，生态广泛，不锁定单一 NIC 厂商
QAIOM / LAN chip 的设计为什么是模块化的？▶
标准答法
Networking uses 2 AIOM modules on OCP 3.0 slots connected to CPU-side PCIe 5.0. This is modular — you choose the NIC speed by selecting the appropriate AIOM card. It keeps the 4 standard PCIe expansion slots free for GPUs or other add-in cards, and avoids locking the board to a specific NIC vendor or speed.

!BMC ≠ Memory Controller。BMC 负责管理服务器，Memory Controller 负责 CPU 和 DRAM 之间的读写，集成在 CPU 内部。
BMC 能做什么
远程开关机、KVM 远程控制台、查看硬件传感器（温度/电源）、挂载虚拟光驱安装系统
Memory Controller 做什么
管理 CPU 和 DRAM 之间所有读写操作，决定内存时序、ECC 纠错、DIMM 通道调度
QBMC 和 Memory Controller 有什么区别？▶
标准答法
They are completely different things. The BMC is a separate ARM processor that handles out-of-band server management — remote power control, KVM access, hardware monitoring — and runs independently even when the main system is off. The Memory Controller is integrated inside the CPU die and handles all data transfers between the CPU and DRAM. One is a management plane, the other is a data path component.

Hardware 面试 Q&A

点击每个问题查看完整回答。这是面试最可能被问到的问题集。

Q1为什么选 dual-socket？能不能用两台单路服务器代替？▶
深度解析
两台单路服务器可以提供总的 CPU 数量，但无法做到内存和数据的物理共享。Dual-socket 的核心优势是两颗 CPU 通过 UPI 共享同一内存地址空间，程序看到的是一台机器的 128 核、8TB 内存。两台独立服务器需要通过网络通信，延迟和带宽无法相比。
Two separate single-socket servers can scale CPU count, but they cannot share memory in a unified address space. A dual-socket system lets both processors access all 8 TB of memory coherently through UPI, which is critical for workloads like large in-memory databases or NUMA-aware applications that need tight coupling between compute and data.
Q2UPI 带宽不够用的时候会发生什么？▶
深度解析
当大量跨 socket 的内存访问发生时，UPI 成为瓶颈，表现为延迟增高、CPU 有时间等待数据。这就是为什么 NUMA-aware 调度非常重要——OS 应该把进程和它的内存放在同一个 NUMA node 上，减少 UPI 流量。可以通过 numactl 命令强制绑定。
If cross-socket memory traffic is high, UPI becomes a bottleneck and latency increases significantly. This is why NUMA-aware scheduling matters — the OS should allocate memory on the same node as the process. On Linux you can use numactl to explicitly bind processes and memory to a specific NUMA node.
Q3PCIe Root Port 是什么？它和 PCIe slot 有什么区别？▶
深度解析
PCIe Root Port 是 CPU 内部的逻辑单元，是 PCIe hierarchy 的根节点，产生实际的 PCIe lanes。PCIe slot 是物理插槽，是 lanes 的物理表现形式。一个 Root Port 可以驱动一个 x16 slot，也可以通过 bifurcation 拆分驱动多个 slot 或设备。
A PCIe Root Port is a logical unit inside the CPU that originates PCIe lanes — it is the root of the PCIe device hierarchy. A physical PCIe slot is just the connector on the board that exposes those lanes to add-in cards. One Root Port can drive a single x16 slot, or be split via bifurcation into multiple narrower logical links.
Q4SAS controller 具体做了什么？为什么 NVMe 不需要？▶
深度解析
SAS controller 做协议转换（PCIe → SAS 协议）、管理 SAS expander 拓扑、提供可选的硬件 RAID。NVMe 不需要这些，因为 NVMe 本身就是基于 PCIe 的协议，host 通过 PCIe 直接和 SSD 通信，CPU 的 NVMe 驱动直接操作设备。NVMe 去掉了 controller 这个中间层。
A SAS controller performs protocol translation from PCIe to SAS, manages SAS expander topologies, and optionally provides hardware RAID. NVMe does not need this because NVMe is a native PCIe protocol — the CPU communicates directly with the SSD over PCIe with no protocol translation. NVMe eliminates the controller intermediary, which is why it has lower latency and higher throughput.
Q5如果需要扩展到 24 个 NVMe 盘，X13DEM 能做到吗？怎么做？▶
深度解析
X13DEM 直连最多 8 个 NVMe (via MCIO)。要扩展到 24 个，需要在 PCIe x16 slot 插一张 NVMe HBA (例如 Supermicro AOC-SLG3-4E2P 或类似产品)，这类 HBA 通过 PCIe x16 接入，内部再 bifurcate 成多路 NVMe 连接。另一种方式是 NVMe-oF (NVMe over Fabrics) 通过网络挂载远端 NVMe。
The X13DEM provides 8 direct NVMe ports via MCIO. To reach 24 drives, you would install a PCIe NVMe HBA in one of the x16 slots — it bifurcates internally to expose additional NVMe connections. Alternatively, NVMe over Fabrics allows mounting remote NVMe storage across high-speed networking, which scales further without consuming local PCIe slots.
Q6IPMI 和 Redfish 的区别是什么？▶
深度解析
IPMI 2.0 是更老的标准，使用二进制协议，命令行工具为 ipmitool。Redfish 是 DMTF 定义的现代 RESTful API 标准，用 JSON 格式，更易于自动化集成（Ansible, Python 脚本等）。AST2600 两者都支持，Redfish 是未来的方向。
IPMI 2.0 is an older binary protocol commonly used with the ipmitool command line. Redfish is a modern RESTful API standard from DMTF that uses JSON, making it much easier to integrate with automation tools like Ansible or Python scripts. Both are supported on this platform, but Redfish is the preferred interface for new automation work.
Q7什么情况下 DMI 4.0 会成为系统瓶颈？▶
深度解析
DMI 4.0 双向约 16 GB/s。如果同时有多个 SATA 盘高速 IO + USB 设备 + M.2 boot 盘大量读写，这些流量都要挤过同一条 DMI 总线。实际场景中，把数据盘都换成 NVMe（走 CPU PCIe）可以彻底避免 DMI 瓶颈，只让 SATA 做低优先级任务。
DMI 4.0 provides about 16 GB/s bidirectionally. If multiple SATA drives are doing heavy I/O simultaneously alongside USB and M.2 traffic, they all share this single DMI link to the CPU. The solution is to use NVMe for high-performance storage, which bypasses DMI entirely by connecting directly to CPU PCIe root ports, leaving DMI only for legacy or low-priority devices.

Linux 框架总览

不要把 Linux 只看成命令集合，要按八个层次来理解，面试里"你会怎么排查"类问题都基于这个框架。

八个核心层次

1. 文件系统
一切都是文件和目录。理解目录结构（/etc, /var, /proc 等）是一切的基础。
2. 用户与权限
谁能读、写、执行。chmod / chown / sudo 是排查权限问题的工具。
3. 进程
程序跑起来就是进程。ps / top / kill 管理进程生命周期。
4. 服务 (systemd)
长期运行的程序由 systemd 管理。systemctl / journalctl 是标准工具。
5. 网络

Q你如何描述你的 Linux 知识框架？▶

When I review Linux, I organize it into filesystem, permissions, processes, services, networking, storage, logging, and shell scripting, because that matches how real troubleshooting usually works. 权限 & 用户

权限模型是 Linux 排查最常触碰的地方。脚本跑不起来、服务读不到文件，第一步查权限。

权限模型

分类含义权限类型
user (owner)文件的所有者r (read=4), w (write=2), x (execute=1)
group文件所属的组
others其他所有人

数字权限速查

# 权限字符串解析
-rwxr-xr--
 rwx      = owner: read + write + execute  = 7
    r-x   = group: read + execute           = 5
       r-- = others: read only              = 4

# 常见权限组合
755 = rwx r-x r-x  (脚本/程序: owner可读写执行, 其他可读执行)
644 = rw- r-- r--  (配置文件: owner可读写, 其他只读)
600 = rw- --- ---  (私钥文件: 只有owner能读写)
777 = rwx rwx rwx  (危险! 所有人都有完全权限)

常用命令

命令功能例子
chmod 755 file设置数字权限使脚本可执行
chmod +x file添加执行权限快速让脚本可运行
chown user:group file修改 owner 和 groupchown nginx:nginx /var/www
whoami当前用户确认自己是谁
iduid / gid / groups 详情查看所属组
sudo以 root 权限执行单条命令sudo systemctl restart nginx

排查权限问题的思路

脚本 Permission Denied
1. ls -la script.sh 查权限
2. 有没有 x 位？
3. 当前用户是 owner / group / others 哪个？
4. 需要的话 chmod +x script.sh
服务读不到文件
1. 服务以哪个用户运行？(ps aux | grep service)
2. 文件 owner 是谁？
3. 该用户有没有 r 权限？
4. 用 chown 或 chmod 修正
Q排查 Permission Denied 的标准流程是什么？▶
标准答法
Linux permissions are based on owner, group, and others, with read, write, and execute bits. In troubleshooting, I first check who owns the file with ls -la, then check what permissions it has, then identify which user the failing process is running as. Usually the fix is either chmod to add the right permission bits, or chown to correct the ownership.

终止进程

命令信号行为什么时候用

kill -9 1234SIGKILL (9)内核强制终止，进程无法拦截进程卡死或拒绝退出时的最后手段

QSIGTERM 和 SIGKILL 有什么区别？实际工作中怎么选？▶
标准答法
For process troubleshooting, I check whether the process exists with ps aux or pgrep, inspect resource usage with top, and if I need to terminate it, I start with SIGTERM to give it a chance to close files and release locks cleanly. SIGKILL is a last resort because the process has no chance to clean up, which can leave stale lock files or corrupt state.

Q外部客户端连不上服务器上的应用，你怎么排查？▶
标准答法
For Linux network troubleshooting, I work through it in layers. First I verify the service is running with systemctl status. Then I check if the port is actually listening with ss -tulpn. If it's listening, I test locally with curl. If that works, the issue is network-level — I check basic connectivity with ping, DNS resolution with nslookup, routing with ip route, and finally whether a firewall rule is blocking the traffic.
Q怎么查某个端口被哪个进程占用？▶
标准答法
I use ss -tulpn — the -p flag shows the process name and PID that owns each socket. For example, ss -tulpn | grep 80 shows everything listening on port 80 along with the process responsible. On older systems without ss, netstat -tulpn provides the same information.

Q应用报错写入失败，怎么快速判断是不是磁盘满了？▶
标准答法
For storage checks, I first run df -h to see filesystem usage across all mount points — if anything shows 100% or close to it, that's the issue. Then I use du -sh to drill down into the largest directories and find what's consuming the space. Common culprits are application logs in /var/log, old database dumps, or core dump files. Disk exhaustion often causes application failures because write operations silently fail.

Linux 面试 Q&A

点击问题查看完整回答。涵盖实际工作中最常被问到的场景题。

Q1你怎么查一个服务有没有起来？▶
完整回答
I start with systemctl status to see if it's active or failed, which also shows the last few log lines. If it's supposed to be running but isn't, I look at journalctl -u to see the full service logs and find the failure reason. I also check ss -tulpn to confirm the service is actually listening on the expected port, because a service can show as active but still not be accepting connections.
Q2你怎么实时看日志？▶
完整回答
For file-based logs, I use tail -f which follows the file and outputs new lines as they appear — the standard way to monitor a live log. I often pipe it: tail -f app.log | grep ERROR to filter noise in real time. For systemd-managed services, journalctl -u servicename -f does the same thing for journal logs.
Q3你怎么查磁盘满了？▶
完整回答
I run df -h first to see the usage percentage for all mounted filesystems. If something is at 90% or above, I use du -sh on the directories under that mount point to narrow down what's consuming space. Common culprits are log files, old database backups, or core dumps. Once I identify the large files, I either archive them, compress them, or set up log rotation to prevent it happening again.
Q4脚本报 Permission Denied，你怎么排查？▶
完整回答
I check three things: first, ls -la on the script to see the permissions — does it have the execute bit set? Second, whoami to confirm which user I'm running as. Third, whether the user is the owner, in the group, or falls under "others" for that file's permissions. If the execute bit is missing, chmod +x fixes it. If the ownership is wrong, chown corrects it. If root permissions are needed, sudo is the solution.
Q5你怎么查某个端口是否在监听，是哪个进程占用的？▶
完整回答
I use ss -tulpn — it shows all listening TCP and UDP sockets with the associated process name and PID. The -p flag is the key one that adds process information. For example, ss -tulpn | grep 8080 immediately tells me whether anything is listening on 8080 and what process owns it. On older systems, netstat -tulpn does the same job.
Q6如何找出某个目录下最近7天内被修改的文件？▶
完整回答
find /path -mtime -7 finds files modified within the last 7 days. The -mtime flag works in 24-hour units, so -7 means "less than 7 days ago." I can add -type f to limit to files only, or -name "*.log" to filter by name. Combining find with xargs is also useful — for example, find /var/log -mtime +30 | xargs ls -lh shows all log files not modified in 30 days that might be candidates for cleanup.
Q7写一个脚本，检查三台服务器是否能 ping 通，打印 UP/DOWN▶
完整代码 
#!/bin/bash

servers=("192.168.1.10" "192.168.1.11" "192.168.1.12")

for host in "${servers[@]}"; do
    if ping -c 1 -W 1 $host >/dev/null 2>&1; then
        echo "$host : UP"
    else
        echo "$host : DOWN"
    fi
done
Key points: -c 1 sends one packet, -W 1 sets a 1-second timeout. >/dev/null 2>&1 discards all output so only our echo statements show. The if statement uses ping's exit code — 0 means success, non-zero means failure.
Q8Linux 里 > 和 >> 的区别是什么？2>&1 是什么意思？▶
完整回答
> redirects stdout and overwrites the file if it exists. >> redirects stdout and appends to the file. 2>&1 means "redirect file descriptor 2 (stderr) to wherever file descriptor 1 (stdout) currently points" — effectively merging error output into the normal output stream. A common pattern is command > output.log 2>&1 which captures both stdout and stderr into one log file.

Class Example

class Sensor:
    def __init__(self, name, unit):
        self.name = name
        self.unit = unit
        self.__reading = 0      # private attribute

    def set_reading(self, value):
        if value < 0:
            raise ValueError("Reading cannot be negative")
        self.__reading = value

    def get_reading(self):
        return self.__reading

    def __str__(self):
        return f"{self.name}: {self.__reading} {self.unit}"


class TemperatureSensor(Sensor):        # Inheritance
    def __init__(self, name):
        super().__init__(name, "°C")

    def is_critical(self, threshold=80):
        return self.get_reading() > threshold


# Usage
s = TemperatureSensor("CPU_Temp")
s.set_reading(92)
print(s)               # CPU_Temp: 92 °C
print(s.is_critical()) # True

Python File Parsing

Reading and parsing output files is a core scripting skill. Covers reading files, splitting/stripping, regex, and structured output parsing.

Reading a File

# Read all lines
with open("output.txt", "r") as f:
    lines = f.readlines()   # list of strings, includes \n

# Iterate line by line (memory-efficient)
with open("output.txt") as f:
    for line in f:
        line = line.strip()  # removes \n and whitespace
        if line:
            print(line)

Parsing Key=Value Output

# Example output.txt:
# Temperature: 72 C
# Fan Speed: 4200 RPM
# Voltage: 12.1 V

data = {}
with open("output.txt") as f:
    for line in f:
        line = line.strip()
        if ":" in line:
            key, val = line.split(":", 1)   # maxsplit=1 for safety
            data[key.strip()] = val.strip()

print(data["Temperature"])  # "72 C"

Parsing with Regex

import re

log_line = "[ERROR] 2024-03-01 CPU temp exceeded threshold: 95 C"

# Extract severity and temperature value
pattern = r"\[(.*?)\].*?(\d+)\s*C"
match = re.search(pattern, log_line)
if match:
    severity = match.group(1)  # "ERROR"
    temp     = match.group(2)  # "95"
    print(f"{severity}: {temp}°C")

# Find all numbers in a string
nums = re.findall(r"\d+\.?\d*", "Fan: 4200 RPM, Volt: 12.1 V")
# ['4200', '12.1']

Parsing JSON / Structured Output

import json, subprocess

Python Interview Q&A

Common Python questions in a hardware/systems engineering interview context.

Q1What is the difference between a list and a tuple?▶
Answer
A list is mutable — you can add, remove, or change elements after creation. A tuple is immutable — once created it cannot be modified. Tuples are slightly faster and are used for fixed data like coordinates or function return values where you want to prevent accidental modification.
Q2Explain inheritance with a real example.▶
Answer
Inheritance lets a child class reuse code from a parent class. For example, a Sensor base class could define common attributes like name and unit, plus a get_reading method. A TemperatureSensor subclass inherits all of that but adds a is_critical() method specific to temperature. You avoid rewriting shared logic and make the code easier to extend.
Q3Write a script to parse a file and extract all lines containing "ERROR".▶
Code 
with open("system.log") as f:
    errors = [line.strip() for line in f if "ERROR" in line]

for e in errors:
    print(e)
Using a list comprehension with a conditional is the Pythonic approach. If the file is very large, it's better to iterate line-by-line rather than loading everything into memory at once — this pattern already does that correctly.
Q4What does "self" mean in a class method?▶
Answer
self refers to the specific instance of the class that the method is being called on. When you write sensor.get_reading(), Python automatically passes the sensor object as the first argument to get_reading. This is how instance methods access and modify the object's own attributes.

IPMI Overview

Intelligent Platform Management Interface — an open standard for out-of-band server management via the BMC. Understand it as the spec that defines how software talks to BMC hardware.

Architecture

BMC (Baseboard Management Controller)
Dedicated microcontroller on the motherboard. Runs independently of the main CPU — works even when the server is powered off (standby power). Implements the IPMI spec.
In-band vs Out-of-band
In-band: host OS talks to BMC via system bus (KCS/SSIF). Out-of-band: management station talks to BMC directly via dedicated LAN (IPMI over LAN / RMCP+).
IPMI Version
IPMI 2.0 is the current standard. Adds encrypted remote access (RMCP+), serial over LAN (SOL), and payload authentication vs IPMI 1.5.
ipmitool
CLI tool to interact with BMC. ipmitool sdr list, ipmitool sel list, ipmitool chassis status are the most common commands.

Key Components

SDRSensor Data RepositoryDatabase of all sensor definitions (type, thresholds, entity)
SELSystem Event LogHardware event log stored in non-volatile BMC memory
PEFPlatform Event FilteringRules engine — filters events and triggers actions
PETPlatform Event TrapSNMP trap sent by BMC when PEF fires an alert action
FRUField Replaceable UnitInventory data for replaceable components (part number, serial)

IPMI Sensor Types & Interfaces

Two core sensor classes defined in the IPMI spec, and the four hardware messaging interfaces BMC uses to communicate with the host.

Sensor Types (IPMI Spec)

KEYAll IPMI sensors fall into two classes: Threshold-based (numeric) or Discrete (event/state bitmask).
ClassHow It WorksExamples
Threshold-basedReturns a numeric reading. Has up to 6 threshold levels: Lower Non-Critical, Lower Critical, Lower Non-Recoverable, and upper equivalents. An event fires when a reading crosses a threshold.Temperature, Voltage, Fan Speed, Current
DiscreteReturns a bitmask of up to 15 event states. Each bit represents a specific condition (asserted/deasserted). No numeric value — just which states are active.Processor status, Power Supply status, Drive Slot state, Button state
Threshold Sensor Thresholds
LNR → LC → LNC → (normal) → UNC → UC → UNR
Lower Non-Recoverable is worst-case low; Upper Non-Recoverable is worst-case high. BMC logs a SEL event when any threshold is crossed.
Discrete Sensor Subtypes
Generic discrete: uses standard event/reading type codes (e.g., 0x03 = state asserted).
Sensor-specific discrete: event meanings are defined per sensor type (e.g., Processor sensor: bit0 = IERR, bit1 = Thermal trip).

BMC Hardware Interfaces (Host-side)

KEYThese are the physical interfaces the host CPU/OS uses to send IPMI messages to the BMC on the same board (in-band).
InterfaceFull NameHow It WorksCommon Use
KCSKeyboard Controller StyleMapped to I/O ports (default 0xCA0/0xCA4). Host writes command bytes to input buffer; BMC processes and writes response to output buffer. Interrupt or polling-based.Most common in-band interface. Used by ipmitool on Linux via /dev/ipmi0.
SMICServer Management Interface Chip3-wire serial interface (SMS_ATN, SMS_DATA, SMS_CLK). Older design, lower bandwidth than KCS.Legacy systems. Rarely seen on modern hardware.
BTBlock TransferSimilar to KCS but with larger buffer size and supports full duplex. Uses 3 I/O registers. Allows sending/receiving bigger payloads in one transaction.Less common; seen in some older server platforms.
SSIFSMBus System InterfaceUses the SMBus (I2C-based) to communicate with BMC. Host acts as SMBus master, BMC as slave. Address 0x10 by default.Used in some embedded/ATCA systems. Also common in BMC-less designs using a dedicated management microcontroller.

PEF / PET / SEL / SDR

How the four components interact: SDR defines sensors → sensors generate events → SEL stores them → PEF filters them → PET delivers alerts.

Component Definitions

ComponentWhat It IsStored In
SDRSensor Data Repository — a table of sensor records. Each record describes one sensor: name, type, entity, thresholds, units, event generation settings.BMC non-volatile storage. Read via ipmitool sdr list.
SELSystem Event Log — a ring buffer of hardware events. Each entry has a timestamp, sensor reference, event direction (assert/deassert), and event data.BMC non-volatile storage. Read via ipmitool sel list.
PEFPlatform Event Filtering — a table of rules. Each rule matches event fields (sensor type, event type, severity) and specifies an action to take when the event matches.BMC volatile/NV storage. Configured via ipmitool pef.
PETPlatform Event Trap — an SNMP trap payload. Sent by the BMC to a configured management station IP when PEF fires an "Alert" action. Carries event data in a standardized format.Transmitted over the network (UDP/162). Received by SNMP trap manager.

QHow does PEF decide to send a PET?▶
Answer
When the BMC receives an event (either from a sensor threshold crossing or a discrete sensor state change), it evaluates the Event Filter Table in PEF. Each filter entry specifies matching criteria — sensor type, event type, event severity. If an incoming event matches a filter entry and that entry's action is set to "Alert", the BMC constructs a PET and sends it as an SNMP trap to the configured destination. The PET payload includes the SEL record data so the management station knows exactly what happened.

Redfish Schema Versions

How to identify which Redfish schema version a URI or property supports — essential for writing compatible management tools.

How to Identify Schema Version

KEYThe primary method is reading @odata.type in any Redfish resource response. It encodes the schema name and version directly.
MethodWhat to Look AtExample
@odata.typePresent in every resource response. Format: #SchemaName.vX_Y_Z.TypeName"@odata.type": "#Thermal.v1_7_0.Thermal" → Thermal schema v1.7.0
Service RootGET /redfish/v1/ — the RedfishVersion field shows the Redfish protocol version"RedfishVersion": "1.16.0"
$metadataGET /redfish/v1/$metadata — OData metadata document listing all schemas and versions supported by this implementationLists all EntityType and EnumType definitions with their namespaces
Schema StoreGET /redfish/v1/schemas/ — collection of schema files served by the BMCEach member is a JSON schema file (e.g., Thermal.v1_7_0.json)

Reading @odata.type

# GET /redfish/v1/Chassis/1/Thermal response:
{
  "@odata.type": "#Thermal.v1_7_0.Thermal",
  "@odata.id":   "/redfish/v1/Chassis/1/Thermal",
  "Id":           "Thermal",
  ...
}

# Breakdown: #SchemaName . vMAJOR_MINOR_ERRATA . TypeName
#   SchemaName  = Thermal
#   Version     = v1_7_0  → 1.7.0
#   TypeName    = Thermal (the specific type within the schema)
TIPWhen writing a management tool that reads Redfish, always check @odata.type before accessing version-specific properties. A property added in v1.5.0 won't exist on a BMC that only supports v1.3.0.
QHow do you check which Redfish schema version is supported without looking at individual resources?▶
Answer
The most complete way is to GET /redfish/v1/$metadata, which returns an OData metadata document listing every schema and version supported by that implementation. For a quick check, GET /redfish/v1/ gives you the RedfishVersion for the protocol version, and individual resources always include @odata.type with the exact schema version for that resource type.

Redfish Telemetry Service

The Redfish Telemetry Service provides a standardized way to collect, define, and report sensor and metric data from a BMC.

Telemetry Schemas Overview

SchemaURI PatternMain Coverage
TelemetryService/redfish/v1/TelemetryServiceEntry point. Exposes links to MetricDefinitions and MetricReportDefinitions collections. Describes max report count, min collection interval, and supported features.
MetricDefinition/redfish/v1/TelemetryService/MetricDefinitions/{id}Defines what a metric is: name, units, data type (Numeric/Boolean/String/Enumeration), implementation (PhysicalContext, sensor type), wildcards for multi-instance metrics.
MetricReportDefinition/redfish/v1/TelemetryService/MetricReportDefinitions/{id}Defines how and when to collect metrics: schedule (interval or OnChange), report type (Periodic/OnChange/OnRequest), which MetricDefinitions to include, and where to log the result.
MetricReport/redfish/v1/TelemetryService/MetricReports/{id}The actual collected data. Contains a timestamped array of MetricValues — each entry has a MetricId, Value, and Timestamp. Generated automatically based on its MetricReportDefinition.
Triggers/redfish/v1/TelemetryService/Triggers/{id}Defines alert conditions on metric values: numeric threshold triggers (Upper/Lower Critical/Warning) or discrete-state triggers. Actions include LogToLogService or sending an event.

Which Schema Covers Sensor Readings?

BMCMetricReport is the primary telemetry source for sensor readings. For direct per-sensor access, the Sensor schema is used.
SchemaSensor Reading CoverageNotes

Q1What is the difference between MetricDefinition and MetricReport?▶
Answer
MetricDefinition describes a metric in the abstract — its name, unit, data type, and what hardware it corresponds to. It doesn't contain any actual readings. MetricReport contains the collected readings — an array of values with timestamps, generated at runtime based on the schedule defined in a MetricReportDefinition. Think of MetricDefinition as the schema and MetricReport as the data.
Q2Which Redfish schema is most relevant to BMC sensor reading design?▶
Answer
For a BMC implementation, the Sensor schema (under /redfish/v1/Chassis/{id}/Sensors/) is the most directly relevant — it gives a per-sensor resource with Reading, ReadingType, thresholds, and health status. For bulk scheduled collection, MetricReport from the TelemetryService is the right answer. In practice, modern BMC designs implement both: Sensor for direct polling and MetricReport for periodic telemetry delivery.
