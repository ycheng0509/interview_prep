Hardware — Core Questions

8 questions most likely to be asked. Each has a one-sentence answer first, then details. Practice saying the EN answer out loud.
最可能被问的 8 道题。每题先给结论句，再展开。把英文答案大声说出来练习。

这一块是 Supermicro 面试最集中考查的区域。8 题全部要熟。面试时先说结论，再展开细节。
Q1 SYS-221HE-FTNR 和 X13DEM 是什么关系？
What is the relationship between SYS-221HE-FTNR and X13DEM? ▶
SYS-221HE-FTNR is the complete 2U short-depth server system — it defines the chassis layout, drive bays, front I/O, and external interfaces. X13DEM is the motherboard inside it, which defines all the core compute capabilities: CPU sockets, memory slots, PCIe lanes, NVMe ports, chipset, and BMC. The same X13DEM board is also used in other Supermicro systems like SYS-121H and SYS-621H.
SYS-221HE-FTNR 是完整的 2U 服务器系统，定义机箱布局、硬盘仓、前置 I/O 和外部接口。X13DEM 是里面的主板，定义所有核心计算能力：CPU Socket、内存槽、PCIe 通道、NVMe 端口、芯片组和 BMC。同一块 X13DEM 主板也用于 Supermicro 其他型号如 SYS-121H。记住：型号 = 系统，X13DEM = 主板，主板定义能力上限。
Q2 为什么这类平台需要双路？单路会在哪些维度受限？
Why dual-socket? What dimensions does a single CPU fall short on? ▶
A dual-socket design is chosen when a single CPU cannot simultaneously provide enough compute, memory capacity, and PCIe I/O bandwidth. For this platform: a single Xeon maxes out at roughly 64 cores, 4 TB memory, and 80 PCIe lanes. Dual-socket doubles all three: 128 cores, 8 TB memory, and ~160 PCIe 5.0 lanes. The key word is "simultaneously" — you need all three at once for workloads like NFV, AI inference, or in-memory databases.
当单颗 CPU 无法同时满足算力、内存容量、PCIe I/O 三个维度的需求时，就选双路。三个维度：1）算力：单路最多 64C → 双路 128C；2）内存：单路 4TB → 双路 8TB；3）PCIe：单路约 80 通道 → 双路约 160 通道。关键词是"同时"——NFV、AI 推理、内存数据库这类 workload 三个维度都需要扩展。面试时从这三个维度展开，不要只说"更快"。
Q3 CPU、Memory Controller、PCIe Root Port、PCH（芯片组）、BMC 各自的职责是什么？
Explain the role of CPU, memory controller, PCIe root port, PCH, and BMC. ▶
The CPU executes instructions and contains the memory controller and PCIe root ports built-in. The memory controller, integrated inside the CPU, directly drives the DDR5 channels. PCIe root ports, also inside the CPU, directly connect to high-bandwidth devices like NVMe SSDs and GPUs. The PCH (Intel C741 chipset) is a southbridge-style chip handling SATA ports, USB, BMC interface, and lower-speed I/O. The BMC (AST2600 on this board) is an independent microcontroller for out-of-band management — it runs even when the host is off, providing IPMI and Redfish access.
CPU：执行指令，内部集成 Memory Controller 和 PCIe Root Port。Memory Controller：在 CPU 内部，直接驱动 DDR5 通道，不是独立芯片，不是 BMC，不是 PCH（这是最常混淆的点）。PCIe Root Port：在 CPU 内部，NVMe 和 GPU 直接连到 CPU 的 Root Port。PCH（Intel C741）：南桥芯片，管 SATA、USB、BMC 接口等低速 I/O。BMC（AST2600）：独立微控制器，做带外管理，主机关机也能运行，提供 IPMI 和 Redfish 访问。
Q4 UPI 是什么？和 QPI 是什么关系？它引出了什么概念？
What is UPI? How does it relate to QPI? What does it introduce? ▶
UPI — Ultra Path Interconnect — is Intel's high-speed cache-coherent socket-to-socket interconnect, running at 16 GT/s with 3–4 links on this platform. QPI was the previous generation, used up through Haswell. The key difference is speed: QPI topped at 9.6 GT/s, UPI 2.0 runs at 16 GT/s. UPI's existence introduces NUMA: because each CPU has its own local memory, accessing the other socket's memory requires crossing UPI, which adds 40–80 ns of latency. The OS scheduler tries to keep processes on their local NUMA node.
UPI（Ultra Path Interconnect）是 Intel 两颗 CPU 之间的高速缓存一致性互联，16 GT/s，3–4 条链路。QPI 是上一代（Haswell 及以前），速率只有 9.6 GT/s。面试时主动说"正确叫法是 UPI，QPI 是老一代"——这体现你知道细节。UPI 引出了 NUMA：每颗 CPU 有自己的本地内存，访问对方内存要跨 UPI，多 40–80ns 延迟。OS 调度器会尽量把进程和内存放在同一 NUMA 节点。
Q5 SATA / SAS / NVMe 的关键差异是什么？
What are the key differences between SATA, SAS, and NVMe? ▶
NVMe connects directly to CPU PCIe lanes via MCIO connectors, delivering up to ~32 GB/s per drive over PCIe 5.0 x4. SATA connects through the PCH chipset, limited to 6 Gbps (~550 MB/s) and further bottlenecked by the DMI link. SAS is not native to this platform — it requires a SAS HBA or RAID controller in a PCIe slot, which acts as a protocol bridge. NVMe is the high-performance path for storage, SATA is for boot media and lower-cost drives, and SAS is for large external JBOD or hardware RAID configurations.
NVMe：直接走 CPU PCIe 通道（MCIO 连接器），PCIe 5.0 x4 每块盘最高约 32 GB/s，延迟极低。SATA：走 PCH 芯片组，限速 6 Gbps（约 550 MB/s），还要经过 DMI link，带宽瓶颈明显。SAS：本平台不原生支持，需要额外装 SAS HBA 卡（占一个 PCIe slot 做协议桥接），适合外部大容量 JBOD 和硬件 RAID。选择逻辑：高性能存储选 NVMe，启动盘/低成本选 SATA，大容量外部存储/RAID 选 SAS。
Q6 什么是 NUMA？对 NIC / NVMe 性能调优有什么影响？
What is NUMA? How does it affect NIC and NVMe performance tuning? ▶
NUMA means each CPU socket has its own local memory bank. Accesses to local memory are fast (one hop), while accessing the other socket's memory requires crossing UPI, adding significant latency. For NICs and NVMe drives, NUMA locality matters because PCIe devices are directly attached to one specific CPU's root complex. For maximum throughput, the application thread, its memory allocations, and the PCIe device should all be on the same NUMA node. Tools like numactl and CPU affinity settings can pin workloads to the correct node.
NUMA：每颗 CPU Socket 有自己的本地内存，访问本地快，跨 Socket 要过 UPI，延迟高约 40–80ns。NIC 和 NVMe 各自直连到某一颗 CPU 的 PCIe Root Port，属于某个 NUMA 节点。调优时要把应用线程、内存分配、PCIe 设备三者都放在同一个 NUMA 节点，避免跨 Socket 访问。工具：numactl --membind 绑内存，taskset 绑 CPU，lscpu/numactl --hardware 查节点拓扑。
Q7 32 DIMM slots 和 2DPC 的性能权衡是什么？
What is the performance trade-off with 32 DIMM slots and 2DPC? ▶
1DPC (1 DIMM per channel) runs at DDR5-5600 with higher memory bandwidth but lower total capacity. 2DPC (2 DIMMs per channel) doubles the DIMM count to 32 slots and enables up to 8 TB total capacity, but the speed drops to DDR5-4400 because the additional DIMM increases electrical load on the channel, forcing a lower clock frequency for signal integrity. Choose 2DPC when the workload is memory-capacity bound (large databases, in-memory analytics); choose 1DPC when memory bandwidth is the bottleneck.
1DPC（每通道 1 根）：速率 DDR5-5600，带宽高，容量低。2DPC（每通道 2 根）：32 个 DIMM 槽，最大 8TB，但速率降至 DDR5-4400。速率下降原因：多一根 DIMM 增加通道电气负载，信号完整性变差，必须降频。选择逻辑：workload 内存容量受限 → 选 2DPC；workload 内存带宽受限 → 选 1DPC。面试时主动说出"速率下降的根本原因是信号完整性"，加分。
Q8 BMC 和 Host OS 分别挂掉时，各自还能做什么？不能做什么？
If the BMC fails vs. if the host OS fails — what can and can't you do? ▶
If the host OS crashes: the BMC remains fully operational. You can still read sensor data via IPMI or Redfish, view the SEL event log, access the remote console (IPMI KVM), power-cycle the host, and update BIOS or BMC firmware — all out-of-band. If the BMC itself fails: you lose all out-of-band management. You can no longer read sensor data remotely, trigger alerts, or access the system if the OS is also down. You'd need physical console access. This is why BMC health monitoring is critical.
Host OS 挂掉时 BMC 仍正常：可以远程读传感器、查 SEL 日志、用 IPMI KVM 访问控制台、远程重启/关机、升级 BIOS/BMC 固件——带外管理全部可用。BMC 挂掉时：失去所有带外管理能力，无法远程读传感器，无法触发告警，如果 OS 也挂了只能物理访问。这就是为什么 BMC 健康监控本身非常重要——BMC 是管理平面的最后防线。

Hardware — Deep Follow-ups

追问题不一定会问，但如果你主动展开这些，会拉开差距。
These deep-dives may not be asked, but volunteering them signals real depth.

追1 某块 NVMe 性能异常，你会先看链路层还是文件系统层？给出排查顺序。
An NVMe drive is underperforming. Link layer first or filesystem first? ▶
Start from the bottom of the stack. 1) Check PCIe link status first: lspci -vv to confirm the NVMe is running at the expected PCIe generation and width (e.g., PCIe 5.0 x4 — if it shows x1 or Gen 3, the link trained down). 2) Check drive health: nvme smart-log /dev/nvme0 for media errors, wear level, and temperature. 3) Check kernel errors: dmesg | grep nvme for PCIe errors, resets, or timeouts. 4) Only then test filesystem or benchmark performance with fio. Reasoning: link problems are hardware-level and explain performance loss more definitively.
从底往上排查，不要先跑 benchmark。1）先查 PCIe 链路：lspci -vv 确认 NVMe 链路速率和宽度是否符合预期（比如 PCIe 5.0 x4，如果显示 x1 或 Gen3 说明 link 降级了）。2）查盘健康：nvme smart-log /dev/nvme0 看 media errors、wear level、温度。3）查内核报错：dmesg | grep nvme 看有没有 PCIe error、reset、timeout。4）最后才用 fio 测性能。排查逻辑：链路层问题是硬件级的，能更明确解释性能下降的根因。
追2 你怎么判断是硬件故障、固件问题，还是 OS 驱动问题？
How do you distinguish hardware failure vs. firmware bug vs. OS driver issue? ▶
Hardware failure: IPMI or Redfish sensor shows abnormal readings (temperature, voltage, error counters); the issue persists across different OS installs or even before the OS boots; you see correctable or uncorrectable PCIe/memory errors in SEL or dmesg. Firmware bug: issue only appears on specific firmware versions; no physical degradation in sensors; may be reproducible with a specific command sequence; often resolved by update. OS/driver issue: issue only appears with a specific kernel version or driver; hardware passes POST and BMC sensors are clean; rolling back the kernel or driver resolves it.
硬件故障：IPMI/Redfish 传感器数值异常（温度、电压、错误计数）；换系统也复现，甚至 POST 前就有；SEL 或 dmesg 有 uncorrectable PCIe/内存错误。固件问题：只在特定固件版本复现；传感器无物理异常；特定操作序列触发；升级固件可解决。OS/驱动问题：只在特定内核版本或驱动复现；硬件 POST 通过，BMC 传感器干净；回滚内核或驱动可解决。区分关键：先看 BMC 传感器（硬件视角），再看是否跨系统复现（固件视角），最后看版本相关性（驱动视角）。

Linux — Core Questions

7 questions. Focus on showing methodology, not just listing commands.
7 道题。重点展示方法论，不只是列出命令清单。

面试官想看的是：你有没有顺序？你知不知道为什么这么做？不要一上来就列 10 个命令，先说"我会先看X，因为Y"。
Q1 你怎么快速确认 CPU / 内存 / 存储 / PCIe 设备拓扑？
How do you quickly confirm hardware topology on a Linux system? ▶
lscpu for CPU cores, sockets, NUMA topology. dmidecode -t memory for DIMM population and speeds. lspci -tv for the PCIe device tree and link widths. lsblk for block devices and their sizes. ip link show for network interfaces. For a comprehensive hardware report: hwinfo or inxi -Fxz. Check NUMA topology with numactl --hardware.
CPU 拓扑：lscpu（核心数、Socket 数、NUMA 节点）。内存配置：dmidecode -t memory（每个 DIMM 槽位置、速率、容量）。PCIe 设备：lspci -tv（设备树和链路宽度）。存储：lsblk（块设备列表和大小）。网络：ip link show。NUMA 拓扑：numactl --hardware。全量硬件报告：hwinfo 或 inxi -Fxz。记忆口诀：CPU用lscpu，内存用dmidecode，PCIe用lspci，磁盘用lsblk。
Q2 服务启动失败，你会按什么顺序排查？
A service fails to start. Walk me through your troubleshooting sequence. ▶
Step 1: systemctl status service-name — get the last error message and exit code immediately. Step 2: journalctl -u service-name -n 50 --no-pager — see the last 50 log lines including startup errors. Step 3: check the unit file systemctl cat service-name — look for ExecStart path, missing dependencies, wrong user. Step 4: check if dependencies are running — systemctl list-dependencies service-name. Step 5: run the binary manually as the service user to see the raw error output. Step 6: check permissions on config files and log directories.
第一步：systemctl status 服务名——立刻看最后一条错误和退出码。第二步：journalctl -u 服务名 -n 50 --no-pager——看完整启动日志。第三步：systemctl cat 服务名——检查 unit file 的 ExecStart 路径、依赖、运行用户。第四步：systemctl list-dependencies 服务名——确认依赖服务都在运行。第五步：手动以服务用户身份运行二进制，看原始错误输出。第六步：检查配置文件和日志目录的权限。顺序逻辑：先看结果（status），再看过程（journal），再看配置（unit），再看依赖，最后动手复现。
Q3 如何定位"端口被占用但应用起不来"的问题？
How do you debug "port already in use but app won't start"? ▶
ss -tlnp | grep :8080 or lsof -i :8080 to find which process is holding the port and its PID. Check if it's a zombie of the same app that didn't clean up. If it's the same app (crashed without releasing the socket), wait for TIME_WAIT to expire or use SO_REUSEADDR. If it's a different process, decide whether to stop it or change the app's port. Always confirm with ss -s for a socket summary.
用 ss -tlnp | grep :端口号 或 lsof -i :端口号 找是哪个进程占用了端口，记下 PID。常见情况：1）同一个应用上次崩溃没释放 socket，处于 TIME_WAIT 状态 → 等待超时或设置 SO_REUSEADDR。2）是另一个不相关的进程 → 决定是停掉它还是改应用端口。3）权限问题：绑定 1024 以下端口需要 root 权限 → 用 setcap 或调高端口。ss -s 查套接字整体统计。
Q4 系统突然变慢：CPU 高、IO wait 高、内存抖动分别看什么？
System is slow. CPU high vs IO wait vs memory pressure — what do you check for each? ▶
High CPU: top or htop to find the process, perf top for call stack profiling. High IO wait: iostat -xz 1 to see which device is saturated, iotop to find which process is doing the I/O. Memory pressure: free -h for overall, vmstat 1 for swap activity (si/so columns), cat /proc/meminfo for detailed breakdown including cached and available. If OOM killer is active: dmesg | grep -i oom.
CPU 高：top/htop 找耗 CPU 的进程，perf top 看调用栈。IO wait 高（CPU 空转等磁盘）：iostat -xz 1 看哪块磁盘饱和（util 接近 100%），iotop 找哪个进程在大量 I/O。内存压力：free -h 看整体，vmstat 1 看 swap 换入/换出（si/so 列有值说明在用 swap），dmesg | grep -i oom 看有没有 OOM killer 杀进程。三种情况的根因不同，不要用同一套命令混查。
Q5 journalctl 和 dmesg 分别适合看什么问题？systemctl enable 和 start 的区别？
journalctl vs dmesg use cases. enable vs start difference. ▶
dmesg shows the kernel ring buffer — hardware events, driver loading, PCIe errors, NVMe resets, memory errors. Best for hardware-level issues and boot-time events. journalctl aggregates logs from systemd and all services with structured metadata (timestamps, service names). Use journalctl -u service -f for service-specific tailing.
For systemctl: start starts the service right now but doesn't survive reboots. enable creates symlinks in the systemd target directories so the service starts automatically on boot, but doesn't start it now. Most of the time you want both: systemctl enable --now service.
dmesg：内核环形缓冲区，看硬件事件、驱动加载、PCIe 错误、NVMe 重置、内存错误。适合硬件级问题和启动时事件。journalctl：聚合 systemd 和所有服务的日志，有结构化元数据（时间戳、服务名）。journalctl -u 服务名 -f 实时跟踪某服务日志。systemctl enable vs start：start = 立刻启动，重启后不自动恢复。enable = 注册开机自启（创建软链接），但现在不启动。通常两个都要：systemctl enable --now 服务名。
Q6 给你一段日志，找过去 30 分钟内所有 ERROR 并按模块计数，怎么做？
From a log file, count all ERRORs in the last 30 minutes grouped by module. ▶
If timestamps are ISO-8601: journalctl --since "30 minutes ago" | grep -i ERROR | awk '{print $5}' | sort | uniq -c | sort -rn. For a raw log file with [module] ERROR format: grep -i ERROR app.log | grep -E "$(date -d '30 min ago' +%H:%M)" | awk -F'[][]' '{print $2}' | sort | uniq -c | sort -rn. The interviewer is testing whether you can combine grep, awk, sort, uniq in a pipeline. Show the pipeline, explain each stage.
如果是 journalctl：journalctl --since "30 minutes ago" | grep -i ERROR | awk '{print $5}' | sort | uniq -c | sort -rn。如果是原始日志文件（格式：[模块] ERROR）：grep -i ERROR app.log | awk -F'[][]' '{print $2}' | sort | uniq -c | sort -rn。管道逻辑：grep 过滤 → awk 提取字段 → sort 排序（uniq 需要相邻行） → uniq -c 计数 → sort -rn 按数量降序。面试官考查的是管道组合能力，解释每一步的作用。

IPMI / Redfish — Core Questions

6 questions. One sentence per concept — don't over-explain. Know the event chain cold.
6 道题。每个概念一句话，不要过度展开。SDR→SEL→PEF→PET 必须背熟。

SDR / SEL / PEF / PET 链是面试必考。 @odata.type 的用途必须能一句话说清楚。Sensor vs TelemetryService 的选择要能解释。
Q1 SDR / SEL / PEF / PET 分别是什么？关系是什么？
What are SDR, SEL, PEF, PET? What is the relationship? ▶
SDR (Sensor Data Record) defines what sensors exist and their thresholds — it's the catalog. SEL (System Event Log) records events when sensor thresholds are crossed — it's the history log. PEF (Platform Event Filter) is a rule engine: when an SEL event matches a rule, PEF decides what action to take (alert, power off, reset). PET (Platform Event Trap) is the delivery mechanism — it sends the alert as an SNMP trap to a remote management system. The chain is: SDR defines → SEL records → PEF decides → PET delivers.
SDR（Sensor Data Record）：定义哪些传感器存在、阈值是多少——是传感器目录。SEL（System Event Log）：当传感器触发阈值时记录事件——是历史日志。PEF（Platform Event Filter）：规则引擎，当 SEL 事件匹配规则时决定做什么（告警/关机/重启）。PET（Platform Event Trap）：交付机制，把告警以 SNMP trap 形式发到远端管理系统。链式关系：SDR 定义 → SEL 记录 → PEF 决策 → PET 外送。这个链条必须能顺口说出来。
Q2 BMC 传感器读值可能经过哪些硬件接口？
What hardware interfaces does the BMC use to collect sensor readings? ▶
The BMC collects sensor data through multiple sideband interfaces: I2C/SMBus for temperature sensors and VRMs, PMBus for power rails and voltage regulators, PECI (Platform Environment Control Interface) for CPU-internal data like die temperature and energy counters, and GPIO for simple on/off signals like fan presence or power button state. These are all out-of-band — they are independent of the PCIe/memory bus that the host CPU uses.
BMC 通过多个 sideband 接口采集传感器数据：I2C/SMBus：温度传感器、VRM（电压调节模块）。PMBus：电源轨电压和功耗数据。PECI（Platform Environment Control Interface）：CPU 内部数据，如 die 温度和能耗计数器，只有 Intel CPU 才有。GPIO：简单开关信号，如风扇存在检测、电源按钮状态。这些都是 out-of-band 接口，独立于主 CPU 使用的 PCIe/内存总线。关键点：BMC 传感器数据来自多种接口，不是单一来源。
Q3 Redfish 的 @odata.type 用来干什么？和 RedfishVersion 有什么区别？
What does @odata.type do in Redfish? How does it differ from RedfishVersion? ▶
@odata.type identifies the schema type and version of a specific resource — for example #Thermal.v1_7_0.Thermal tells you this is a Thermal resource conforming to schema version 1.7.0. It's on every resource. RedfishVersion (from /redfish/v1) tells you the version of the Redfish protocol/specification the service implements, such as "1.16.0". The difference: @odata.type is per-resource schema versioning (each resource type evolves independently), while RedfishVersion is the service-level protocol version. Use @odata.type to know which properties are available on a specific resource.
@odata.type：标识某个具体资源的 schema 类型和版本，例如 #Thermal.v1_7_0.Thermal 表示这是 Thermal 资源，符合 schema v1.7.0。每个资源都有。用途：知道这个资源有哪些属性可用。RedfishVersion（来自 /redfish/v1）：整个 Redfish 服务实现的协议/规范版本，如 "1.16.0"。区别：@odata.type 是资源级 schema 版本（每种资源类型独立演进），RedfishVersion 是服务级协议版本。面试一句话：@odata.type 告诉你这个资源"是什么版本的什么类型"，RedfishVersion 告诉你这个 Redfish 服务"支持哪个版本的协议"。
Q4 拉温度/电压/风扇数据并告警，选 Sensor 还是 Telemetry schemas？为什么？
To pull temperature/voltage/fan data and trigger alerts — Sensor or TelemetryService? ▶
For direct real-time readings and threshold-based alerts: use the Sensor schema under /redfish/v1/Chassis/{ChassisId}/Sensors/{SensorId}. Each sensor has Reading, ReadingType, ReadingRangeMin, ReadingRangeMax, and health status. Good for polling and immediate checks. For scheduled bulk collection, historical aggregation, or building a monitoring pipeline: use TelemetryService schemas — MetricDefinition defines what to collect, MetricReportDefinition schedules the collection, MetricReport delivers the data, and Triggers can fire on thresholds. In modern BMC implementations you'd use both: Sensor for on-demand checks, TelemetryService for continuous monitoring.
Sensor（直接读值）：/redfish/v1/Chassis/{id}/Sensors/{id}，每个传感器有 Reading、ReadingType、阈值范围和健康状态。适合直接 polling 和即时告警检查。TelemetryService（监控管道）：MetricDefinition 定义采集什么，MetricReportDefinition 定义采集计划，MetricReport 交付数据，Triggers 定义阈值触发告警。适合周期性批量采集、历史数据和监控系统集成。现代 BMC 两者都实现：Sensor 用于按需查询，TelemetryService 用于持续监控管道。
Q5 IPMI 和 Redfish 在自动化和可扩展性上的权衡是什么？
IPMI vs Redfish for automation and scalability — trade-offs? ▶
IPMI uses a binary command protocol (KCS/SSIF/LAN) with raw command codes. It's widely supported and works on virtually all servers, but it's not human-readable, hard to extend, and the tooling (ipmitool) uses command-specific syntax that doesn't compose well into modern automation pipelines. Redfish is REST+JSON, which integrates naturally with HTTP clients, Python requests, Ansible, and CI/CD pipelines. It's schema-documented, versioned, and supports richer data models. Trade-off: IPMI has universal hardware support and lower overhead; Redfish is the modern standard for automation and is where new features are being developed.
IPMI：二进制命令协议（KCS/SSIF/LAN），用 raw command code，基本所有服务器都支持，但不可读、难扩展，ipmitool 命令格式各异，不适合现代自动化管道。Redfish：REST+JSON，自然集成到 HTTP 客户端、Python requests、Ansible、CI/CD，schema 有文档、有版本、数据模型更丰富。权衡：IPMI 硬件覆盖更广，开销更小；Redfish 是现代自动化标准，新功能只在 Redfish 上开发。实际工作中两者共存：IPMI 做兼容性保障，Redfish 做新自动化工具。

Fan1         | N/A   | cr
             |       |          ← 空行/坏行
Voltage Rail | 12.1  | ok
Fan2         | 3200  | ok
             | 55    |          ← 字段不完整
Write parse_sensor_file(path) → list[dict] with fields name/value/status.

# ── 题 A: 传感器文件解析 ──────────────────────────────────────────
# EN: Parse a pipe-delimited sensor file, clean bad data, return records + summary
# 中: 解析管道分隔的传感器文件，清洗坏数据，返回记录列表和摘要

def parse_sensor_file(path: str) -> tuple[list[dict], int]:
    """
    Returns (records, invalid_line_count).
    records: list of {'name': str, 'value': str, 'status': str}
    中文: 返回 (有效记录列表, 无效行数)
    """
    records = []
    invalid_count = 0

    with open(path, 'r', encoding='utf-8') as f:
        for line in f:                        # 逐行读 — 不把整个文件加载到内存
            line = line.strip()              # 去首尾空白 / strip whitespace
            if not line:                     # 跳过空行 / skip blank lines
                continue

            parts = [p.strip() for p in line.split('|')]

            if len(parts) != 3:            # 必须有 3 个字段 / must have exactly 3 fields
                invalid_count += 1
                continue

            name, value, status = parts

            if not name:                     # 名称不能为空 / name must not be empty
                invalid_count += 1
                continue

            records.append({
                'name': name,
                'value': value,
                'status': status
            })

    return records, invalid_count


def summarize(records: list[dict], invalid_line_count: int = 0) -> dict:
    """
    Produce a summary of the parsed records.
    中文: 对解析结果生成摘要
    """
    # 大小写无关匹配 / case-insensitive status check
    bad_items = [
        r['name'] for r in records
        if r['status'].lower() != 'ok'
    ]

    return {
        'total_count': len(records),
        'bad_count': len(bad_items),
        'bad_items': bad_items,
        'invalid_line_count': invalid_line_count
    }


# ── 使用示例 / Usage ──────────────────────────────────────────────
if __name__ == '__main__':
    records, invalid = parse_sensor_file('sensors.txt')
    result = summarize(records, invalid)
    print(result)

# ── 题 B: 日志 ERROR 聚合 ─────────────────────────────────────────
# EN: Count ERROR occurrences per module, return top N by count
# 中: 统计每个模块的 ERROR 次数，返回 Top N

import re
from collections import defaultdict

# 匹配 [模块名] 后跟 ERROR（大小写无关）
# EN: matches [module_name] followed by ERROR (case-insensitive)
_PATTERN = re.compile(r'\[(\w+)\]\s+ERROR', re.IGNORECASE)


def top_error_modules(lines: list[str], n: int = 3) -> list[tuple[str, int]]:
    """
    Returns list of (module_name, error_count) sorted by count descending.
    Skips malformed lines silently.
    中文: 返回 (模块名, 错误数) 列表，按错误数降序，跳过格式错误的行
    """
    counts: dict[str, int] = defaultdict(int)

    for line in lines:
        m = _PATTERN.search(line)   # search 比 match 更灵活 — 可在行中任意位置匹配
        if m:                         # 没匹配到就跳过（坏行、WARN 行等）
            counts[m.group(1)] += 1  # group(1) 是 (\w+) 捕获的模块名

    # 排序：按 count 降序，取前 n 个
    # EN: sort by count descending, take top n
    return sorted(counts.items(), key=lambda x: -x[1])[:n]


# ── 使用示例 / Usage ──────────────────────────────────────────────

    sample = [
        "2026-03-24T10:15:02Z [network] ERROR link down",
        "2026-03-24T10:15:05Z [sensor]  WARN  high temp",
        "2026-03-24T10:15:10Z [network] ERROR packet loss",
        "2026-03-24T10:15:12Z [disk]    ERROR read timeout",
        "malformed line",
    ]
    print(top_error_modules(sample, n=3))
    # → [('network', 2), ('disk', 1)]

# ── 题 C: Redfish 温度告警过滤 ───────────────────────────────────
# EN: Extract overheat sensors from a Redfish JSON response
# 中: 从 Redfish JSON 提取超温传感器列表

from datetime import datetime, timezone
from typing import Any


def find_overheat(
    readings: list[dict[str, Any]],
    threshold: float = 80.0
) -> list[dict]:
    """
    Filter sensor readings above threshold.
    Returns list of {sensor_name, reading_celsius, severity, timestamp}.
    中文: 过滤超过阈值的传感器读值，返回结构化告警列表
    """
    alerts = []
    now = datetime.now(timezone.utc).isoformat()

    for sensor in readings:
        # .get() 防 KeyError — Redfish JSON 字段不一定都存在
        name = sensor.get('Name', 'unknown')
        raw_reading = sensor.get('ReadingCelsius')

        if raw_reading is None:        # 没有读值，跳过 / skip missing readings
            continue

        try:
            reading = float(raw_reading)  # 类型转换加保护 / guarded type conversion
        except (ValueError, TypeError):
            continue                      # 坏数据静默跳过 / skip bad data

        if reading > threshold:
            # 严重程度：> threshold+15 = critical，否则 warning
            severity = 'critical' if reading > threshold + 15 else 'warning'
            alerts.append({
                'sensor_name': name,
                'reading_celsius': reading,
                'severity': severity,
                'timestamp': now
            })

    return alerts


# ── Mock 数据示例 / Mock data ─────────────────────────────────────
if __name__ == '__main__':
    mock_readings = [
        {'Name': 'CPU1 Temp',  'ReadingCelsius': 92.5},  # critical
        {'Name': 'CPU2 Temp',  'ReadingCelsius': 83.0},  # warning
        {'Name': 'Inlet Temp', 'ReadingCelsius': 35.0},  # ok
        {'Name': 'Disk Temp',  'ReadingCelsius': 'N/A'}, # bad data — skipped
        {'Name': 'Fan Sensor'},                           # missing field — skipped
    ]
    for alert in find_overheat(mock_readings, threshold=80.0):
        print(alert)
